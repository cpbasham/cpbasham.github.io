Arrays and Hashes

A Quick Intro

6/7/15

Arrays

Arrays in Ruby are ordered collections of objects. They are indexed by integers counting up from 0.

Brackets are used both for indexing and for representing/printing arrays:
arr = ["zero", "one", "two", "three"]
puts arr[2] => "two"
You can throw any data types into Ruby arrays, including more arrays!
arr = [0, 1, 2.2, :three, ['another', 'array'], "FIVE"]
fifth_element = arr[4] # fifth_element = ['another', 'array']
To create an empty array, you can use empty brackets:
empty = []
Alternatively, you can use an array constructor. Using the constructor you can even automatically fill the array with a set value:
empty = Array.new => []
zeros = Array.new(5, 0) => [0, 0, 0, 0, 0]
There are some other cool things about arrays, but I'll leave them for a different post.

Hashes

Hashes are ordered collections of objects as well, but their ordering is not as simple as that of arrays and is generally disregarded. Instead hashes are used because the index (known as a key) can be any object, rather than just an integer.

Just like in arrays, brackets are used for indexing in hashes. However, when being represented/printed, hashes are wrapped in curly braces and the key-value pairs are shown together, connected by '=>':
my_hash = {:gringo => true, 99 => "problems", "age" => 22.5} puts my_hash["age"] => 22.5 puts my_hash[:gringo] => true
As partially demonstrated above, you can put any type of object in a hash as either the key or the value.
To create an empty hash, you can use empty curly braces or a constructor:
empty = {}
empty = Hash.new # equivalent
Also, hashes come with a default value of nil (returned when a value is requested for a non-existent key). You can change this by explicitly setting the default or by specifying this in the constructor:
my_hash = Hash.new("DEFAULT")
# or after the hash is created...
my_hash.default = "DEFAULT"
puts my_hash["nonexistent key"] => "DEFAULT"

So why would you choose to use one over the other? Well, part of the tradeoff is speed vs flexibility. Arrays are generally more efficient/faster than hashes, at the expense of only using integer indices. Hashes take a slight performance hit in order to gain more possible usages. In the end, if you need an ordered list or you see another good reason to use an array, then definitely stick to the array. But don't force it. I know from experience that the tiny gains from using arrays aren't worth the lack of readability brought on by trying to force functionality into the wrong data structure.
